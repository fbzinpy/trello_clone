Guía de estudio para parcial en vivo. Ordenada de fácil a difícil. Pensada para saber qué tocar, por qué tocarlo y cómo resolver bajo presión.

1. Flujo sagrado

Request HTTP ↓ ROUTE → define verbo y URL ↓ MIDDLEWARE → valida token / valida body ↓ CONTROLLER → extrae datos de req, llama service, responde ↓ SERVICE → lógica pura con Prisma ↓ DATABASE
Regla madre
  • Si hay una URL nueva, casi siempre tocas route + controller + service.
  • Si hay un campo nuevo en base de datos, casi siempre tocas schema.prisma + schema Zod + controller + service.
  • Si hay seguridad, revisa middleware y el uso de req.user.
  • Si solo cambia un texto, un orden o una condición, probablemente basta con service o controller.

Patrón Qué significa Archivos típicos Dificultad

4. Fácil

Patrón 1

Editar lógica existente

Qué hace: cambia algo que ya existe sin crear endpoints nuevos ni tocar la estructura general.

Cómo afrontarlo: busca la función que ya hace algo parecido y modifica solo la línea clave. No reescribas todo.

// Ejemplo mental
// "cambiar expiración del token"
// → buscar jwt.sign(..., { expiresIn: '7d' })
// → reemplazar solo ese valor
  • Cambiar expiración del JWT.
  • Cambiar mensajes de error o éxito.
  • Cambiar orden de resultados con orderBy.
  • Quitar o cambiar columnas por defecto.
  • Agregar un dato al return.
  • Limitar cantidad con count() + if + throw new Error().

Cheatcode de límites

const funcion = async (foreignId, data) => {
  const count = await prisma.model.count({ where: { foreignId } })
  if (count >= LIMITE) {
    throw new Error('Mensaje del límite')
  }

  return prisma.model.create({
    data: { foreignId, ...data }
  })
}

Cheatcode de validación sencilla

if (!campo?.trim()) {

5. Medio

Patrón 2

Endpoint nuevo

Qué hace: agrega una operación nueva como editar, obtener uno por id, renombrar, etc.

Cómo afrontarlo: piensa en la secuencia service → controller → route. Siempre empieza por el service porque es la parte más concreta.

// SERVICE
const nuevaFuncion = (id, dato) =>
  prisma.model.update({
    where: { id },
    data: { campo: dato }
  })

// CONTROLLER
const nuevaFuncion = async (req, res) => {
  try {
    const { dato } = req.body
    if (!dato?.trim()) return res.status(400).json({ error: 'Dato requerido' })
    const result = await service.nuevaFuncion(Number(req.params.id), dato.trim())
    res.json(result)
  } catch (e) {
    res.status(500).json({ error: e.message })
  }
}

// ROUTE
router.patch('/ruta/:id', validate(schema), c.nuevaFuncion)
  • Editar título de card.
  • Editar nombre de board.
  • Editar label de column.
  • Obtener board por id.
  • Obtener cards de una columna.
  • Duplicar un board.

Cheatcode para GET por id con ownership

// SERVICE
const getById = (id, userId) =>
  prisma.board.findFirst({
    where: { id, userId },
    include: {
      columns: {
        include: { cards: { orderBy: { order: 'asc' } } },
        orderBy: { order: 'asc' }
      }
    }
  })

// CONTROLLER
const getById = async (req, res) => {
  try {
    const item = await boardService.getById(Number(req.params.id), req.user.id)
    if (!item) return res.status(404).json({ error: 'No encontrado' })
    res.json(item)
  } catch (e) {
    res.status(500).json({ error: e.message })
  }
}

6. Medio-alto

Patrón 3

Campo nuevo en la base de datos

Qué hace: agrega una propiedad como description, color, priority, dueDate, username.

Cómo afrontarlo: sigue siempre el mismo orden. No improvises.

  1. Agregar el campo en schema.prisma.
  2. Actualizar el schema de Zod si entra por request.
  3. Actualizar el service para guardar o editar el campo.
  4. Actualizar el controller para leerlo desde req.body.
  5. Si es necesario, actualizar la route o reutilizar la existente.
// schema.prisma
model Card {
  id          Int     @id @default(autoincrement())
  title       String
  description String?
  columnId    Int
  column      Column  @relation(fields: [columnId], references: [id], onDelete: Cascade)
  order       Int     @default(0)
}
// zod
const addCardSchema = z.object({
  title: z.string().min(1).max(200),
  description: z.string().max(500).optional()
})
// service
const addCard = (columnId, title, description) =>
  prisma.card.create({
    data: {
      columnId,
      title,
      description: description || null
    }
  })
  • Agregar description a Card.
  • Agregar color a Column.
  • Agregar priority a Card.
  • Agregar dueDate a Card.
  • Agregar username a User.
  • Agregar isArchived a Board.

Comando mental de Prisma

npx prisma migrate dev --name nombre_del_cambio

Si el profe te pregunta teóricamente, di: “edito schema.prisma, hago migración, actualizo service/controller/schema Zod”.

7. Difícil

Patrón 4

Seguridad, roles y ownership

Qué hace: decide si un usuario autenticado puede o no hacer algo.

Cómo afrontarlo: piensa siempre en req.user. Si el token ya fue verificado, ahí viven id, email y role.

// route protegida por rol
router.get('/admin/users', verifyToken, authorizeRoles('admin'), authController.getAllUsers)
// ownership en service con filtros
const deleteBoard = (id, userId) =>
  prisma.board.deleteMany({ where: { id, userId } })
// ownership en controller si necesitas buscar primero
const algo = await service.getById(Number(req.params.id))
if (!algo) return res.status(404).json({ error: 'No encontrado' })
if (algo.userId !== req.user.id) return res.status(403).json({ error: 'Sin permiso' })
  • Solo admin puede ver todos los usuarios.
  • Solo dueño puede borrar board.
  • Solo dueño puede ver board por id.
  • Bloquear creación si usuario está suspendido.
  • Permitir acceso a varios roles con authorizeRoles('admin', 'user').

8. Casos que más probablemente te pasen

Caso Nivel Qué tocar Cómo pensarlo
Cambiar expiración JWT Fácil auth service Buscar jwt.sign y reemplazar expiresIn.
Cambiar mensajes Fácil controller Buscar res.json o res.status(...).json.
Limitar cantidad de boards/cards Fácil-media service count() → if → throw → create().
Editar card/board/column Media service + controller + route + schema si entra body Pensar en update() con id.
Traer un item por id Media service + controller + route findFirst o findUnique + 404 si no existe.
Agregar campo nuevo Media-alta prisma + zod + controller + service Seguir orden fijo, no saltarte pasos.
Validar rol o dueño Difícil middleware y/o service/controller Pensar en req.user y filtros por userId.

9. Plantillas rápidas

Plantilla controller

const funcion = async (req, res) => {
  try {
    const { campo } = req.body
    if (!campo?.trim()) return res.status(400).json({ error: 'Campo requerido' })

    const result = await service.funcion(Number(req.params.id), campo.trim())
    res.json(result)
  } catch (e) {
    res.status(500).json({ error: e.message })
  }
}

Plantilla service update

const funcion = (id, campo) =>
  prisma.model.update({
    where: { id },
    data: { campo }
  })

Plantilla zod

const schema = z.object({
  campo: z
    .string({ required_error: 'Campo obligatorio' })
    .min(1, 'No puede estar vacío')
    .max(100, 'No puede superar 100 caracteres')
})

10. Errores típicos que te pueden hacer perder tiempo

  • Olvidar async cuando agregas un await.
  • Olvidar convertir req.params.id a número con Number().
  • Poner comas entre métodos encadenados de Zod.
  • Usar findUnique con filtros que no son únicos.
  • Olvidar exportar la nueva función en module.exports.
  • Crear la función en controller pero no agregar la route.
  • Agregar route y controller pero olvidar el schema de validación.
  • Usar throw('mensaje') en vez de throw new Error('mensaje').

11. Mantra de parcial

Si me piden cambiar algo: primero identifico si es lógica, endpoint, campo o seguridad.

Si es lógica: busco la función y toco lo mínimo.

Si es endpoint: hago service, luego controller, luego route.

Si es campo: prisma, zod, service, controller.

Si es seguridad: pienso en middleware, req.user y userId.
